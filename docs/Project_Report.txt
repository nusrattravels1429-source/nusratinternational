ABSTRACT
================================================================

The Nusrat International Information System is a full-stack
web application developed to digitize and modernize the
operations of Nusrat International, a licensed travel and
manpower recruitment agency based in Dhaka, Bangladesh
(BAIRA RL-1429).

The system replaces the traditional paper-based record-keeping
approach with a centralized, cloud-connected digital platform.
It enables administrators to manage travel packages, Hajj/Umrah
packages, work visa packages, gallery items, team members,
certifications, hero sliders, and client messages — all through
a secure admin dashboard. Clients can browse all packages,
contact the agency, and view certifications through a fully
responsive, multilingual (English & Bengali) public website.

The backend is built on Node.js and Express, the database is
MongoDB (hosted on MongoDB Atlas), images are managed via
Cloudinary, and the platform is deployed on Vercel.

1. INTRODUCTION
================================================================

1.1. INTRODUCTION OF PROJECT
-------------------------------
The Nusrat International Information System is a comprehensive
web-based platform designed to serve as the official digital
presence and backend management system for Nusrat
International — a travel, Hajj/Umrah, and manpower recruitment
agency in Bangladesh.

The system keeps a structured digital record of all travel and
work packages, processes client inquiries, manages gallery
images, team profiles, and certification documents, and allows
the agency to update all website content in real time through a
secure administrative dashboard.

Key highlights of the system:
- Dynamic package management (Travel, Hajj, Work)
- Hero slider & homepage content management
- Team member and certification management
- Client message inbox
- Multilingual content support (English & Bengali)
- SEO-optimized public pages
- Cloud image storage via Cloudinary
- Deployed publicly on Vercel

1.2. AIM OF PROJECT
---------------------
The existing system relies entirely on manual, paper-based
record keeping with no centralized digital system. The aim of
this project is to computerize all the informational and
operational activities of Nusrat International using a modern
web-based Information System.

The main module of the system is the Administrative Module,
through which a single authorized administrator can manage all
aspects of the organization's digital content and client
communications.

1.3. SYSTEM STUDY & ANALYSIS
------------------------------
System analysis is the process of gathering and interpreting
facts, diagnosing problems, and recommending improvements.
It requires intensive communication between system users
and system developers.

During the analysis phase, detailed interviews were conducted
with the agency staff to understand the daily operational
workflows. The findings were used to design a system that
closely mirrors real operational processes, while eliminating
inefficiencies.

================================================================
2. EXISTING SYSTEM
================================================================

2.1. EXISTING MANUAL SYSTEM
-----------------------------
Currently, Nusrat International operates entirely on a
paper-based system. Package details, client records, visa
processing statuses, and team information are maintained in
physical files and registers. Images and certifications are
stored in printed formats in filing cabinets.

2.2. PROCESS OF EXISTING SYSTEM
---------------------------------
- Packages are described in printed brochures.
- Client inquiries are handled over phone/in-person only.
- Certification records are maintained in physical folders.
- Team information has no formal public record.
- There is no way for clients to browse packages online.

2.3. PROBLEMS WITH EXISTING SYSTEM
------------------------------------
- Lack of data security (files can be lost or damaged).
- Time-consuming manual searches for records.
- High paper usage and storage requirements.
- No 24/7 accessibility for clients.
- No centralized tracking of client messages.
- Difficult to update package information quickly.
- No scalability for handling high volumes of inquiries.
- No analytics or statistical summary available.

3. PROPOSED SYSTEM
================================================================

3.1. AIM OF PROPOSED SYSTEM
-----------------------------
The proposed system aims to fully digitize and centralize the
operations of Nusrat International by providing:
- A public-facing website for client engagement.
- A secure admin dashboard for content management.
- Cloud-based image storage for all media assets.

- Significant reduction in manual work and paperwork.

- Output:  Dynamic web pages, JSON API responses, admin views

Logical flow:
  Client Browser --> Express Router --> Controller --> MongoDB
       |                                                  |
       <------------- EJS Template (rendered HTML) --------

---------------------
The physical design defines the actual deployment architecture:

  [Client Browser]
       |
  [Vercel CDN / Serverless Function]
       |
  [Express.js App (app.js)]
       |
  [Route Layer] --> /admin, /api, /
       |
  [Controller Layer] --> cardController, authController, etc.
       |
  [MongoDB Atlas] <--> [Cloudinary Storage]

- Server: Vercel Serverless (Node.js 22.x)
- Database: MongoDB Atlas (Cloud, nusrat_travels DB)
- Static Assets: Vercel CDN (/public directory)
- Images: Cloudinary CDN

5. IMPLEMENTATION OF MODEL
================================================================

5.1. ANALYSIS MODELING & DESIGN METHODOLOGIES
-----------------------------------------------
This project uses structured analysis methodology. The system
has very few transformation flows (data is mostly transactional
— create/read/update/delete) making structured analysis the
most appropriate choice over OOA.

The analysis model achieves three objectives:
1. Describes what the admin and client users do.
2. Establishes a basis for the system design.
3. Defines requirements that can be validated after build.

The core of the model is the MongoDB schema (data dictionary),
surrounded by:
- Entity Relationship conceptual model (collections & refs)
- Data Flow Diagram (request/response cycles)

5.2. DATABASE DESIGN
---------------------
MongoDB (NoSQL document database) is used. The database name
is "nusrat_travels". Collections are used instead of tables.

The design follows two levels:
1. Information Level Design: What data is needed and how it
   relates (collection schemas, field types, relationships).
2. Physical Level Design: MongoDB Atlas configuration, indexes,
   connection pooling, and serverless-optimized caching.

Data integrity is enforced via:
- Mongoose schema validation (type checking, required fields)
- Unique slug indexes on the `cards` collection
- isActive / isPaused flags for soft-delete patterns

5.3. ENTITY RELATIONSHIP MODEL
--------------------------------
MongoDB uses a document model rather than relational tables.
Relationships are handled via referenced _id fields or embedded
subdocuments. The conceptual ER model maps closely to the
collection structure.

5.4. IDENTIFYING THE ENTITIES (COLLECTIONS)
---------------------------------------------
The system uses the following MongoDB collections:

1.  admins          - Administrator accounts
2.  cards           - Travel / Hajj / Work packages
3.  galleryitems    - Photo gallery images
4.  teammembers     - Staff and founder profiles
5.  certifications  - Agency certifications & licenses
6.  sitecontents    - Homepage hero text, page intros, etc.
7.  messages        - Client contact form submissions (inbox)
8.  navlinks        - Navigation link configuration
9.  footersettings  - Footer content configuration

5.5. ENTITY RELATIONSHIP DIAGRAM (INTRODUCTION)
-------------------------------------------------
ERD enables the software engineer to fully specify data objects,
their attributes, and their relationships.

Key terms:
- Entity: A MongoDB collection (e.g., cards, teammembers)
- Attribute: A field within a document (e.g., title, slug)
- Relationship: A reference from one collection to another
  (e.g., a card's imageUrl references a Cloudinary asset)
- Primary Key: MongoDB's auto-generated `_id` (ObjectId)
- Reference: An ObjectId field pointing to another collection

5.6. RELATIONSHIP CARDINALITY
-------------------------------
- admins      --> sitecontents  : 1:N (admin manages many)
- admins      --> cards         : 1:N (admin manages many)
- cards       --> images[]      : 1:N (one card, many images)
- teammembers --> image         : 1:1 (one member, one photo)
- certifications --> certImage  : 1:1

5.7. DATABASE COLLECTION STRUCTURE
------------------------------------

COLLECTION: admins
  _id        : ObjectId (Primary Key)
  username   : String (required, unique)
  password   : String (hashed with bcryptjs)
  isActive   : Boolean
  createdAt  : Date

COLLECTION: cards
  _id        : ObjectId (Primary Key)
  type       : String (enum: 'travel', 'hajj', 'work')
  title      : { en: String, bn: String }
  slug       : String (unique, auto-generated)
  description: { en: String, bn: String }
  images[]   : [{ url, caption, order }]
  location   : { en: String, bn: String }
  pricing    : { amount, currency, displayText }
  duration   : String
  features[] : [{ en: String, bn: String }]
  isActive   : Boolean
  isPaused   : Boolean
  order      : Number
  createdAt  : Date
  updatedAt  : Date

COLLECTION: galleryitems
  _id        : ObjectId (Primary Key)
  title      : String
  imageUrl   : String (Cloudinary URL)
  category   : String
  isActive   : Boolean
  order      : Number
  createdAt  : Date

COLLECTION: teammembers
  _id        : ObjectId (Primary Key)
  name       : String
  role       : String
  email      : String
  imageUrl   : String (Cloudinary URL)
  isFounder  : Boolean
  isActive   : Boolean
  order      : Number

COLLECTION: certifications
  _id        : ObjectId (Primary Key)
  name       : String
  certImage  : String (Cloudinary URL)
  issuedBy   : String
  isFeatured : Boolean
  isActive   : Boolean
  order      : Number

COLLECTION: sitecontents
  _id        : ObjectId (Primary Key)
  section    : String (e.g., 'homepage', 'about', 'hajj')
  key        : String (e.g., 'homepage-hero-text')
  title      : String
  content    : String
  imageUrl   : String
  order      : Number

COLLECTION: messages
  _id        : ObjectId (Primary Key)
  name       : String
  email      : String
  phone      : String
  subject    : String
  message    : String
  isRead     : Boolean
  createdAt  : Date

   - Full CRUD for all content types
   - Multi-image upload via Cloudinary
   - Hero slider management with reordering
   - Real-time dashboard statistics
   - Client message inbox with read/unread status

5.9. DATA FLOW DIAGRAM (DFD)
-----------------------------

LEVEL 0 - CONTEXT DIAGRAM:

  [Client] ---request---> [Nusrat International System] ---data--> [MongoDB Atlas]
     |                             |
     <----response (HTML/JSON)-----+-----> [Cloudinary] (images)

LEVEL 1 - MAIN PROCESSES:

  1. Authentication Process
     Admin --> Login Form --> JWT Verify --> Dashboard

  2. Package Management Process
     Admin --> Create/Edit Form --> Multer (upload) -->
     Cloudinary --> MongoDB (cards collection) --> Public Pages

  3. Content Management Process
     Admin --> Section Form --> sitecontents collection -->
     Homepage / Travel / Hajj / Work pages

  4. Client Inquiry Process
     Client --> Contact Form --> messages collection -->
     Admin Inbox --> Mark Read / Delete

  5. Gallery Management Process
     Admin --> Image Upload --> Cloudinary -->
     galleryitems collection --> /gallery page

6.3. USING GUIDELINE

FOR CLIENTS (Public Website):
  1. Visit the website URL.
  2. Browse packages via navigation: Travel, Hajj, Work.
  3. Click any package card for full details.
  4. Use the Contact page to submit inquiries.
  5. Visit About, Gallery, and Certifications for more info.

FOR ADMINISTRATOR (Admin Dashboard):
  1. Navigate to /admin/login.
  2. Enter admin username and password.
  3. Upon login, you are redirected to /admin/dashboard.
  4. Dashboard shows live statistics for all content types.

  Managing Packages:
  - Go to Admin > Cards
  - Click "Add New Card" to create a Travel/Hajj/Work package
  - Fill title (EN/BN), description, location, price, duration
  - Upload up to 4 images
  - Click Save; the package appears on the public website

  Managing Gallery:
  - Go to Admin > Gallery
  - Upload images with title and category
  - Toggle active/inactive status

  Managing Team:
  - Go to Admin > Team
  - Update founder profile or add/edit/remove team members

  Managing Certifications:
  - Go to Admin > Certifications
  - Add certification name, issuing body, and certificate image
  - Toggle "Featured" to show on homepage

  Managing Inbox:
  - Go to Admin > Inbox
  - View client messages submitted via the contact form
  - Mark as read or delete messages

  Managing Site Content:
  - Go to Admin > Content sections (Homepage, Hajj, Travel, etc.)
  - Update intro text, hero text, and section images

  Logout:
  - Click "Logout" in the sidebar to end the session securely
